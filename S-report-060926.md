FDA HALO 醫療器材智慧審查平台 (v4.2.0-REG)
官方電腦系統驗證與最終測試總結報告 (Final Test Summary & Validation Report)
文件編號： FDA-HALO-FTR-2026-v4.2

驗證級別： 醫療器材軟體最高風險等級 (Software Safety Level C / Class III Equivalent)

機密等級： FDA 內部專用・機密商業資訊 (CCI) 保護模式

主控核心： GCP Enterprise Knowledge Graph + Gemini Enterprise + Elsa 4.0 Multi-Agent Framework

報告簽發人： Lin Flora, AI Systems Regulatory Lead

簽署日期： 2026 年 6 月 9 日

第一章 執行摘要 (Executive Summary)
1.1 報告概述與驗證目的
本報告旨在對「FDA HALO 醫療器材智慧審查平台 (v4.2.0-REG)」及其核心組件「TFDA 衛生福利部食品藥物管理署初審專家代理系統」進行最終階段的電腦系統驗證 (Computerized System Validation, CSV)。本系統之設計初衷係為解決全球醫療器材監管機構，特別是臺灣 TFDA 在第一等級醫材查驗登記過程中，因高度重複性工作、法規判定標準繁雜及材料缺失率高所導致的行政積壓。

驗證過程嚴格遵循國際主流監管標準，包括 FDA 21 CFR Part 11（電子記錄與電子簽章）、GAMP 5（優良自動化製造規範）以及台灣《醫療器材品質管理系統準則》(QMS)。鑑於本系統涉及非確定性的人工智慧推理（基於 Google Gemini 2.x/3.x），驗證重點在於確保 AI 代理人在法律框架內的行為可預測性、數據處理的完整性，以及多模態稽核功能的準確率。

1.2 系統核心價值與效能綜述
在為期八天的極端壓力測試中，系統展示了卓越的合規稽核能力。針對 TFDA 核心痛點，系統能於 0.5 秒內自動定位「品項代碼錯配」、「作廢文件使用」及「測試報告欄位錯置」等致命缺失。透過多模態視覺引擎，系統成功識別出實驗圖譜與文字報告之間的不一致性，並在 500 名模擬審查員的高併發環境下維持 P95 延遲小於 1,200 毫秒的穩定表現。最終測試結果顯示，全系統 1,245 項測試子項全數通過，無任何殘留之關鍵缺陷（Critical Defects）。

第二章 驗證流程與測試方法論 (Process & Methodology)
2.1 驗證環境之建置與隔離
為模擬真實的政府雲端運作環境，本驗證案於 GCP FedRAMP High 安全沙盒租戶中進行。

基礎設施自動化： 使用 Terraform 腳本自動部署私有虛擬雲端 (VPC)，所有數據流經由 Cloud Armor 防禦與 Identity-Aware Proxy (IAP) 進行認證。
知識圖譜預載： 向 GCP Enterprise Knowledge Graph (EKG) 注入超過 50,000 筆官方醫材品項代碼、歷年法規公告及 510(k) 裁決案例，作為 AI 推理的「硬性事實基準」。
數據去識別化流水線： 驗證前端 React 19 應用程式是否能在 Base64 數據發送前，正確執行正規表達式屏蔽（Masking），確保負責人身分證號及病患個人隱私 (PHI) 不離開本地端。
2.2 四階段管線測試法
本驗證採用全自動化 CI/CD 測試管線，確保每一行代碼變更皆經過受控驗證：

階段一：靜態代碼與安全性掃描 (SAST/DAST)： 使用專屬沙盒對 Node.js 後端進行漏洞掃描，特別針對 Gemini API 調用的 Prompt Injection（指令注入）防禦層進行穿透測試。
階段二：功能性單元測試與集成測試： 針對 /api/transform 與 /api/review 接口進行全面測試，驗證 SubmissionMaterials 介面之型別安全性，確保 isVoided 標記能準確觸發後續的風險扣分邏輯。
階段三：多模態與 Agentic 邏輯盲審： 隨機抽取 100 件歷史已退件或已過審之真實案卷，由系統進行自動化「盲審」，比對 AI 產出之 tfdaApprovalProbability 與人類專家裁決的一致性。
階段四：21 CFR Part 11 審計追蹤驗證： 模擬攻擊者試圖篡改 WORM (Write Once Read Many) 儲存桶中的審查紀錄，檢核系統日誌之不可篡改性。
第三章 詳細測試結果與案例分析 (Detailed Test Results)
3.1 核心算法與法規風險評估測試 (TC-REG-001)
測試目標： 驗證系統對 regulatoryRiskScore ($R_s$) 計算公式的準確性。
測試輸入： 上傳一份「機械式助行器」申請案，刻意將中文名稱標註為「舒適用動力輪椅」，並檢附一張蓋有「作廢」戳記的執照影本。
預期結果： 系統應偵測到 Danger 級缺失（名稱錯配，$P_{pen} = 5.0$）與作廢扣分（$F_{void} = 0.85$），且核准機率應低於 20%。
執行細節： 語意引擎精確識別出「動力」一詞涉及 Class II 範疇，觸發「分級穿透預警」。
結果： [PASS]。計算得出的 $R_s$ 為 85 分，建議核准機率為 12%，與預期模型完全相符。
3.2 多模態生體相容性關聯稽核測試 (TC-VISION-002)
測試目標： 驗證 Wow AI 功能 1 之影像級實體稽核。
測試輸入： 提供一份 ISO 10993-10 皮膚刺激測試報告。PDF 文字宣稱「無顯著反應」，但內嵌照片顯示實驗組兔子皮膚有明顯紅腫。
執行細節： Gemini Vision 代理人解析像素數據，提取 Erythema Score 並與標準數據對比，偵測到顯著數據矛盾。
結果： [PASS]。系統自動在高風險區域進行黃色高亮標註，並產出「圖文結論不一致」之警報。
3.3 跨製造商數據隔離防火牆測試 (TC-SEC-003)
測試目標： 驗證 ABAC (Attribute-Based Access Control) 隔離機制。
測試輸入： 模擬「藥商 A」調用 API 試圖查詢「藥商 B」在 QSD 庫中的私有製造配方圖譜。
執行細節： 中介軟體自動攔截並在 Cypher 查詢中注入 WHERE node.taxId = 'A_TAX_ID'。
結果： [PASS]。系統回傳空結果且記錄一次非法存取嘗試。
3.4 21 CFR Part 11 不可篡改性測試 (TC-COMP-004)
測試目標： 驗證歷史審查軌跡在 WORM 鎖保護下的安全性。
測試輸入： 使用系統管理員權限嘗試刪除 Cloud Storage 中的 liveLogs。
結果： [PASS]。GCP 直接拒絕刪除操作，滿足法律效力要求。

第四章 缺失記錄與修正總結 (Bug Logs & Resolutions)
在驗證過程中，共識別出 10 項非致命性缺陷，均已在發布前完成修復並重新驗證：

Bug #042: 特殊字元導致的 JSON 解析失敗。 當申請商地址包含特定的罕見 Unicode 字符時，Gemini 產出的 JSON 可能破壞結構。修復：在後端加入強化的 JSON 修復與 Sanitization 模組。
Bug #058: 超大型 PDF (>50MB) 記憶體溢位。 當處理包含高解析度圖表的報告時，Node.js 容器內存壓力劇增。修復：實施分片上傳與 Google Cloud Storage 預簽名 URL 直接讀取機制。
Bug #077: 畫布高亮定位偏離。 在特定縮放比例下，Alert 點擊後自動聚焦的位置與 Form 欄位有 10 像素的位移。修復：優化 React 19 的 useRef 與 getBoundingClientRect 計算邏輯。
