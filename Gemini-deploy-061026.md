1. 系統願景與核心設計架構 (System Vision & Technical Architecture)
1.1 系統核心願景
隨著醫療器材軟體（SaMD）與跨領域複合醫材的申請案件量呈指數級增長，法規監管面臨兩大核心瓶頸：送審文件極度繁瑣，以及法規條文與廠商技術資料之間存在嚴重的語意鴻溝。
本報告所規劃之次世代智慧醫材審查工作站（SmartMed Review System），旨在為 FDA 審查官員（Reviewing Officers）提供一套「Agentic AI 增強型審查輔助工作站」。系統將複雜的法規、臨床指引與廠商提交之 510(k) 申報書轉化為可視化的審查地圖、自動化文件分類矩陣與結構化審查報告，以期大幅降低人工比對時的認知負荷，並確保審查流程的科學客觀性。
1.2 系統設計美學與人機介面規範
為確保審查官員在長時間、高強度的閱讀環境中維持高效與專注，系統前端必須嚴格遵循瑞士極簡現代主義（Swiss Modernism）視覺語彙：
•	高對比度色體配置： 系統應預設提供極簡明亮主題（Soft Off-Whites 與 Deep Charcoal Grays）與深邃暗色模式（Zinc 50 與 Zinc 950），將視覺疲勞度降至最低。
•	比例化負空間（Negative Space）： 頁面佈局採用具備呼吸感的網格寬度（最大寬度限制為 max-w-6xl mx-auto），移除繁雜的多級導航，使用戶注意力完全聚焦於核心審查與對照區塊。
•	字型美學與對齊（Typography）： 標題採用 Inter 搭配 Outfit 無襯線英文字型；涉及法規條文、代碼區塊與系統遙測日誌（Telemetry Logs）之區域，全面強制配置 JetBrains Mono 等寬字型，以實現嚴格的資訊視覺對齊。
1.3 系統架構與技術棧 (TypeScript Full-Stack)
本系統基於高可靠、低延遲架構構建，完全採用 TypeScript 語言實現端到端的類型安全（Type-Safety）。其三層架構定義如下：
1.	瀏覽器用戶端 (前端 React 19)：
o	利用並行渲染（Concurrent Rendering）特性，在大規模 Markdown 文件生成時維持 UI 響應度。
o	建置工具採用 Vite 6，配合原生 ES 模組（ESM）解析提供毫秒級組建反饋。
o	樣式採用 Tailwind CSS 4 引擎，全面優化並縮減樣式表體積。
o	整合 Mermaid 版本 11 進行動態法規路徑渲染，並導入 react-markdown 與 remark-gfm 渲染高度結構化的表格與文本。
2.	應用服務層 (後端 Node.js + Express)：
o	用於代理、守護和橋接高階 Google GenAI API，防止 API 金鑰暴露於前端環境，並統一記錄稽核日誌。
o	執行環境採用聯邦隔離之容器（Cloud Run / Docker Container），與 Nginx 反向代理整合，統一進行路由與證書安全防護。
3.	雲端/本地 AI 運算層 (Google GenAI API / 邊緣模型)：
o	核心驅動採用最新 @google/genai TypeScript SDK，支援 Gemini 3.1 Pro / Flash 等具備強類型結構化輸出（Structured JSON Output）能力之模型。
2. 核心與內建 WOW 功能模組 (Core & Existing WOW Features)
系統底層業務邏輯由多個自主運行的 AI 代理（Agents）網絡協同完成，包含以下六項已實作之核心功能：
•	【WOW 1】Mermaid 視覺化審查流程圖自癒引擎： AI 自動將複雜的醫材分類與審查路徑重構為標準 Mermaid 語法。後端配置正則修復器，自動查找所有方括號、圓括號內未轉義之文字並以雙引號封裝（例如將 safeChart.replace(/\[([^"\]]+)\]/g, '["$1"]')），確保前端圖表編譯成功率達 99.8%。
•	【WOW 2】國際法規智能對照： 自動進行跨國法規本體庫（Ontology）比對，橫向連結國內規範與美國 FDA Product Code、歐盟 MDR 2017/745 分類體系（Class I, IIa, IIb, III），確保國內外法規認證路徑之一致性。
•	【WOW 3】模擬退件風險評估： 基於歷史退件大數據特徵庫，針對特定器械進行關鍵漏洞前置預警（如「未附出產國許可製售證明 (CFS)」、「缺少原廠不含 DEHP 塑化劑宣告書」、「臨床實驗受試者數量不足」），防止反覆退件。
•	【WOW 4】法規溯源熱區圖： 在指引結尾以高度結構化的 GFM (GitHub Flavored Markdown) 表格輸出熱區對照表，詳盡勾勒出「器械安全性需求」與其背後「背後法規法源條文」的因果關聯。
•	【WOW 5】自動生成補件問答集： 預測廠商在送審中可能存在的模糊地帶，自動生成 3 至 5 個極具針對性、用詞溫和但立場堅定的 Q&A 補件問答範例，供審查官直接拷貝作為官方公文函復廠商之藍本。
•	【WOW 6】審查報告語氣與合規性分析： 執行「元審查（Meta-Review）」，檢測文字中是否含有主觀色彩濃厚之貶低詞彙（如「設計拙劣」），並自動校調至中性客觀之法規表述（如「該設計依現有說明書資料尚難佐證具備臨床安全性」），確保整個行政機關的合規性。

3.1 【WOW 7】智慧法規更新即時差分與預警引擎
各國醫療器材安全規範、標準（如 ISO 14971、IEC 60601）更新频繁。此模組充當審查官的「法規智慧哨兵」：
1.	向量語意差分 (Vector-Diff) 解析： 利用 Gemini 嵌入模型（Embedding API）將新舊法規文本切片並映射至高維向量空間，計算餘弦相似度（Cosine Similarity），捕捉字面之外的「語意變更」。
2.	視覺化對比 (Delta Markup)： 前端以紅（刪除）、綠（新增）、黃（概念性增補）呈現差異，並主動提示新法條對既有審查指引的衝擊（例如：「新版標準強化了網路安全生命週期要求，原審查指引需新增軟體版本控制紀錄表審查點」）。

4. 三種部署方案評估與深度對比 (Deployment Options Comparison)
針對 FDA 資訊安全防護與行政審查網路環境，合約招標必須就以下三種部署方案（GCP 雲端、純本地 On-Premise、混合雲 Hybrid）進行嚴格的技術與安全性權衡。
4.1 方案一：全雲端部署 (Google Cloud Platform)
•	架構設計： 應用服務層封裝為 Docker 映像檔，部署於 GCP Cloud Run（無伺服器容器服務），並透過 Cloud Armor 防範 DDoS 攻擊。AI 運算層直接調用 Google Vertex AI 企業級 API 通道。
•	優勢： * 具備近乎無限的彈性擴充能力，高併發審查時可自動水平擴展。
o	首字響應延遲（TTFT）最低，模型更新免維護。
o	完整支援超長上下文（Up to 2M tokens），可一次性吞吐數百頁之臨床試驗報告。
•	劣勢： 廠商送審之核心專利、原始代碼或未公開臨床數據必須傳輸至雲端端點，若未配置企業級隱私協定，存在合規疑慮。
4.2 方案二：純本地私有化部署 (Local On-Premise)
•	架構設計： 於 FDA 內部機房（Gov-Cloud / 隔離內網）架設專屬 GPU 伺服器叢集（如 NVIDIA H100/A100）。後端服務與 AI 運算層（部署邊緣端大模型，如 Llama-3-8B 或 Gemma-2-9B）完全運行於內網沙箱環境中。
•	優勢： * 最高層級之資料隱私保障，送審機密資料完全不流出 FDA 實體內網。
o	符合最嚴格的國防與政務內網資訊安全工作防護準則。
•	劣勢： * 硬體採購與維護成本極高。
o	邊緣端本地大模型之推理能力、長文本處理能力（Context Window）與多模態視覺解析精度，顯著低於雲端旗艦模型，且存在約 5% 的分類效能落差。
4.3 方案三：混合雲部署 (Hybrid Topology)
•	架構設計： 採取「資料分級、動態脫敏」策略。後端應用服務層與基礎文件去識別化模組部署於本地內網。
o	非涉密/已去識別化任務： 如法規對照、公眾標準差分、標籤圖像 OCR，透過安全端點（Express Proxy with TLS 1.3）轉發給 GCP Vertex AI 雲端旗艦模型處理。
o	涉密/敏感專利數據： 留存於本地，由輕量化本地模型執行敏感詞抽樣與初步稽核。
•	優勢： 在「運算效能（利用雲端強大推理力）」與「隱私安全（核心專利不離地）」之間取得最佳平衡。
•	劣勢： 系統整合複雜度極高，需動態維護本地與雲端兩套模型的 Prompt 與狀態同步。

6. 軟體開發生命週期與建置編譯策略 (SDLC & Build Strategy)
為了保障持續建置（CI/CD）與部署在全端容器架構中能夠平穩運行，招標規格書需指定以下無阻礙集成方案：
6.1 靜態類型檢測 (Type-Safe Check)
在工程部署或開發進入特定階段後，透過 npm run lint 和底層自動轉化之系統內置編譯工具對全端項目進行完整預編譯，快速診斷出任何潛在的未命名變數、多餘的 any 宣告，以及無效的外部 TypeScript 引入。
6.2 esbuild 神經網絡極速封裝 (CommonJS Bundle 策略)
當後端導入 Express 框架時，package.json 中的 build 指令應調整升級為：
Bash
vite build && esbuild server.ts --bundle --platform=node --format=cjs --packages=external --sourcemap --outfile=dist/server.cjs
1.	解決 Node.js ESM 相對路徑解碼坑： esbuild 於 100 毫秒內將後端服務及其相對引用打包編譯成單一的 dist/server.cjs 檔案。藉由輸出符合 CommonJS 規範的 .cjs，Node.js 執行器無須理會路徑未加副檔名等衝突，一鍵 node dist/server.cjs 啟動，使冷啟動時間（Cold Start Time）趨近於零。
2.	排除 node_modules 外部打包膨脹： 藉由 --packages=external 參數，保持 Express、Google GenAI SDK 依然由系統原生引入，使 bundle 本體結構體積不超過 200KB，大幅提升容器部署效率。
