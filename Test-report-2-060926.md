1. 導論與驗證目標 (Introduction & Validation Objectives)
1.1 背景說明
SmartMed 2.0 系統（研發代號：FDA HALO）是一套基於高階生成式人工智慧（Gemini 3.1 Pro/Flash）與企業級知識圖譜（GCP EKG）構建的智慧醫材審查輔助平台。鑑於本系統直接參與 TFDA/FDA 510(k) 醫療器材實質等同性（SE）之裁決輔助，其軟體缺陷可能導致具備臨床風險的醫材錯誤獲准上市，或造成高度敏感之廠商商業秘密（CCI）外洩。因此，本計劃書旨在定義一套符合國際法規標準的電腦系統驗證（CSV）框架。

1.2 驗證目標
本計劃的核心目標在於確保系統在「非確定性 AI 模型」與「確定性法律邏輯」之間取得合規平衡。具體目標包括：

確保數據隔離： 驗證不同醫材廠商間的資料在圖數據庫與 AI 脈絡中具備 100% 的物理或邏輯隔離。
消除 AI 幻覺： 透過 Elsa 4.0 多 Agent 矩陣與確定性圖譜對應，確保所有法規結論皆有據可循。
合規性核稽： 確保系統完整符合 21 CFR Part 11 關於電子記錄與電子簽章的不可篡改性要求。
性能穩定性： 驗證系統在高負載審查期間（如 500 併發）仍能維持低延遲響應。
2. 參照標準與合規性矩陣 (Regulatory Compliance Matrix)
本驗證計劃嚴格遵循以下國際與國內標準：

FDA GAMP 5 (A Risk-Based Approach to Compliant GxP Computerized Systems)： 作為系統生命週期管理的頂層架構。
IEC 62304 / IEC 82304-1： 針對醫療器材軟體生命週期過程與健康軟體系統的安全要求。
21 CFR Part 11： 針對電子記錄（Electronic Records）與電子簽章（Electronic Signatures）的技術控管。
ISO 14971： 醫療器材風險管理在軟體演算中的應用。
TFDA 智慧醫材審查指引： 確保系統輸出之報告符合國內行政程序法與專業審查語境。

4. 驗證策略與方法論 (Validation Strategy)
4.1 基於風險的驗證方法 (Risk-Based Approach)
本系統採用「V-Model」結合「敏捷開發驗證」模式。針對高風險功能（如法規判定、數據隔離），執行 100% 的測試覆蓋；針對低風險功能（如畫家 Persona 語氣變換），執行代表性抽樣測試。

4.2 三階驗證架構 (IQ / OQ / PQ)
安裝驗證 (IQ)： 驗證 Cloud Run 容器、EKG 實例、KMS 加密金鑰及 VPC 服務控制之配置正確性。
操作驗證 (OQ)： 驗證各功能模組（如 Mermaid 渲染、JSON 剖析）在極端參數下的邊界反應。
性能驗證 (PQ)： 模擬真實審查場景，驗證多 Agent 在處理數千頁技術文件時的邏輯穩定性與合規輸出率。

6. 核心測試案例設計 (Detailed Test Case Design)
6.1 安全防禦類：Text-to-Cypher 注入攔截 (TC-SEC-001)
測試目的： 防止攻擊者透過 Prompt 指令惡意操作圖數據庫。
方法： 模擬輸入包含 DROP TABLE 或 MATCH (n) DELETE n 語義的自然語言。
預期結果： AST 解析器應精準攔截，並在執行前觸發 403 異常，不對圖譜造成任何實體變動。
6.2 數據隔離類：跨廠商 ABAC 權限測試 (TC-COMP-002)
測試目的： 驗證 A 廠商的審查員絕對無法查看到 B 廠商的機密演算參數。
方法： 以 B 廠商帳戶登入，並指定查詢 A 廠商特定 K 號案件。
預期結果： 系統應自動在 Cypher 查詢中注入 WHERE sponsor_id = 'B'，回傳結果應為空或僅限公共資訊。
6.3 AI 邏輯類：法規差分與更新預警測試 (TC-WOW7-003)
測試目的： 驗證當 ISO 14971 從 2012 版更新至 2019 版時，系統能準確辨識「風險管理計畫」之強制性變更。
方法： 上傳新舊版本 PDF，啟動 Vector-Diff 引擎。
預期結果： 系統應以紅色/綠色標註語義變更點，並自動產出對現有審查指引的修改建議清單。
6.4 多模態類：標籤合規性視覺稽核 (TC-WOW8-004)
測試目的： 驗證系統能識別標籤上字體大小是否低於 6pt。
方法： 上傳包含 4pt 微小字體警告語的彩色標籤圖檔。
預期結果： 系統應在 Canvas 上以紅框標註該區域，並引用《醫療器材標籤包裝管理辦法》判定不合規。
7. 軟體生命週期與變更控制 (SDLC & Change Control)
7.1 持續驗證 (Continuous Validation)
鑑於 LLM 模型會定期進行更新，本系統建立「黃金數據集（Golden Dataset）」迴歸管線。每當底層 Gemini 模型發生版本更迭（如從 1.5 升級至 3.0），系統將自動對 100 件歷史標竿案件進行盲審。若符合率低於 99.5%，則自動阻斷部署。

7.2 變更影響分析 (Impact Analysis)
任何提示詞（Prompt）的微調皆被視為軟體變更，必須經過版本控制（Git Tagging）與影響分析報告（Impact Report）。重大變更需經審查委員會（Review Board）核准後方可推送到測試環境。

8. 數據安全與隱私保護機制 (Data Security)
8.1 零留存政策 (Zero-Retain Policy)
系統與 Google Enterprise API 簽署免於訓練協定。所有審查官上傳的原始文件在處理完成後，其內存快取將被即刻抹除。

8.2 加密傳輸與存儲
傳輸中數據 (Data in Transit)： 強制使用 TLS 1.3。
存儲中數據 (Data at Rest)： 使用 AES-256 加密，金鑰由 GCP KMS 管理並定期進行人工輪轉驗證。
9. 允收基準與失敗處理 (Acceptance Criteria & Failure Management)
9.1 硬性允收指標
Critical Defects: 0 (如數據隔離失效、系統崩潰)。
Major Defects: < 2 (如特定語氣顯示錯誤，但不影響法規判定)。
測試覆蓋率: 單元測試 > 95%，法規決策路徑 100%。
9.2 失敗處理程序
若測試失敗，系統將進入「冷封存」狀態。開發團隊需提交根因分析（RCA）報告，修復後需重新執行受影響模組的所有相關測試案例，確保無回歸風險。

10. 驗證總結與後續行動 (Conclusion & Future Actions)
本計劃書為 SmartMed 2.0 的正式驗證基準，旨在建立一個可信、透明且具備法律效力的 AI 輔助審查環境。隨著 3.0 版本的開發，我們將進一步探討「聯邦學習」在不觸碰廠商原始數據前提下的模型優化驗證。

6. 核心測試案例設計 (Detailed Test Case Design)
6.1 安全防禦類：Text-to-Cypher 注入攔截 (TC-SEC-001)
測試目的： 防止攻擊者透過 Prompt 指令惡意操作圖數據庫。
方法： 模擬輸入包含 DROP TABLE 或 MATCH (n) DELETE n 語義的自然語言。
預期結果： AST 解析器應精準攔截，並在執行前觸發 403 異常，不對圖譜造成任何實體變動。
6.2 數據隔離類：跨廠商 ABAC 權限測試 (TC-COMP-002)
測試目的： 驗證 A 廠商的審查員絕對無法查看到 B 廠商的機密演算參數。
方法： 以 B 廠商帳戶登入，並指定查詢 A 廠商特定 K 號案件。
預期結果： 系統應自動在 Cypher 查詢中注入 WHERE sponsor_id = 'B'，回傳結果應為空或僅限公共資訊。
6.3 AI 邏輯類：法規差分與更新預警測試 (TC-WOW7-003)
測試目的： 驗證當 ISO 14971 從 2012 版更新至 2019 版時，系統能準確辨識「風險管理計畫」之強制性變更。
方法： 上傳新舊版本 PDF，啟動 Vector-Diff 引擎。
預期結果： 系統應以紅色/綠色標註語義變更點，並自動產出對現有審查指引的修改建議清單。
6.4 多模態類：標籤合規性視覺稽核 (TC-WOW8-004)
測試目的： 驗證系統能識別標籤上字體大小是否低於 6pt。
方法： 上傳包含 4pt 微小字體警告語的彩色標籤圖檔。
預期結果： 系統應在 Canvas 上以紅框標註該區域，並引用《醫療器材標籤包裝管理辦法》判定不合規。
