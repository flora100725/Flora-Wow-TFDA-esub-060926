1. 系統願景、瑞士美學與本地化核心哲學 (System Vision & Philosophy)
1.1 系統核心願景
隨著軟體即醫療器材（SaMD）及搭載複雜人工智慧演算法之影像診斷工具的送審量呈爆發性成長，法規主管機關面臨前所未有的審查高壓。送審文件動輒數千頁，法規條文與廠商自主研發之底層技術數據之間存在巨大的「語意鴻溝」。
SmartMed 2.0 (SmartMed Review 2.0-060326) 智慧醫材審查指引與清單生成系統，旨在為審查官員提供一套部署於內部隔離網路的「本地化 AI 增強型審查輔助工作站」。本系統整合了可在本地高效能運行的 Gemini-3.1-Flash-Lite 輕量敏捷模型架構，將枯燥的法規條文、歷史退件特徵與廠商提交之 510(k) 申報書轉化為可視化的審查地圖、強類型文件分類矩陣與結構化審查報告，旨在極大化降低人工比對時的認知負荷，同時 100% 杜絕涉密智慧財產權外洩。
1.2 瑞士極簡現代主義介面規範
為確保審查官員在長時間、高強度的閱讀環境中維持最高專注度，系統前端必須嚴格遵循瑞士極簡現代主義（Swiss Modernism）視覺語彙：
•	高對比度底色配置： 系統應預設提供優雅的極簡明亮主題（Soft Off-Whites 與 Deep Charcoal Grays）和深邃暗色模式（Zinc 50 與 Zinc 950），確保在任何環境光線下眼睛疲勞度降至最低。
•	比例化負空間（Negative Space）： 頁面佈局完全捨棄繁雜的多級導航欄與干擾視線的進度條，主視窗全面配置具備呼吸感的網格寬度限制（max-w-6xl mx-auto），使主管官員的注意力集中於核心審查區塊。
•	字型美學與資訊對齊： 標題採用高科技質感的 Inter 搭配 Outfit 無襯線英文字型；涉及法規條文、程式碼區塊與系統遙測日誌的區域，全面配置 JetBrains Mono 等寬字型，以實現嚴格的資訊視覺對齊。
•	微動效（Micro-Animations）： 利用原生優化之 motion 庫進行平滑的主題轉換與分頁渲染，消除頁面切換時的生硬閃爍感。

•	建置工具： Vite 6。高效率的開發伺服器，配合原生 ES 模組（ESM）解析，提供毫秒級的組建與熱模組替換（HMR）反饋。
•	樣式解決方案： Tailwind CSS 4。全新 CSS 變數編譯引擎，在編譯期進行高度最佳化，大幅縮減樣式表體積至 45KB 以內。
•	核心動畫與圖表庫： 採用 motion/react 次級包進行無縫分頁轉折；整合 Mermaid 版本 11 實現動態法規路徑渲染；可視化圖形庫配置 D3.js 與 Recharts 作為高階數據可視化媒介。
•	資料解析組件： react-markdown 與 remark-gfm，用以渲染高度結構化的 Markdown 文本、公式與標準化法規表格。
2.2 後端微服務與邊緣 AI 核心
•	後端引擎： Node.js 與 Express。部屬至本地獨立實體伺服器，統一監聽 Port 3000，負責代理、守護和橋接本地 AI 運算層，防止 API 密鑰與端點在外網暴露，並統一記錄完整的行政稽核日誌（Audit Logs）。
•	AI 核心驅動（預設 Gemini-3.1-Flash-Lite 架構）： 系統在本地 GPU 伺服器上，透過 vLLM 或 TensorRT-LLM 高效能推理引擎，常駐載入與 Google 官方 Gemini-3.1-Flash-Lite 具備同等敏捷度、強類型 JSON 輸出能力之邊緣端大型語言模型。後端使用官方最新 @google/genai TypeScript SDK，配置內部底層端點（baseURL）重定向至本地私有 IP 網段（例如 http://10.20.30.40:8000/v1），徹底切斷一切外網連接。
3. 系統需求規格書項目項目剖析 (Detailed SRS Items)
為確保政府合約採購與建置具備高度的工程約束力，以下將系統功能以標準系統需求規格（SRS）規格進行量化定義。
3.1 功能需求規格 (Functional Requirements)
3.1.1 智慧指引生成模組 [SRS-F-001]
•	功能描述： 系統必須允許審查官員上傳 .txt 或 .md 格式的異質法規原始草案，或者由審查官手動貼上法規核心文本，系統將自動重構為標準化審查指引。
•	串流響應效能 [SRS-F-001-1]： 系統全面整合 generateTextStream 推理架構。透過 for await...of 非同步迭代器，即時將本地模型輸出的權杖（Tokens）以字元流形式渲染於瀏覽器介面上，首字響應時間（TTFT）在本地萬兆網路環境下必須小於 350 毫秒。
•	畫家擬真風格設定（Painter Personas）[SRS-F-001-2]： 系統必須內建多元思考維度切換器。透過將 System Instruction 微調為「達文西（精準解剖與力學拆解）」、「梵谷（醫學人文關懷與患者舒適度）」、「林布蘭（Revolutionary 功效與失效風險的強烈明暗對比）」、「畢卡索（多維解構法規要求）」等世界大師的藝術哲學，將枯燥條文重構為具備穿透力的論述，引導官員從不同維度審視器材設計。
•	非同步中斷機制 [SRS-F-001-3]： 前端必須配置精緻的深色遙測終端機（Telemetry Terminal）。當審查官按下「立即中斷 (Abort)」功能時，系統必須透過對前端傳遞的 AbortSignal 進行監聽，立刻向本地推理伺服器發送 TCP 終止框架，全面中斷本地顯示卡算力消耗。
3.1.2 送審清單智慧分類與強類型分撿模組 [SRS-F-002]
•	功能描述： 系統必須提供雙向大塊文本對照視窗。左側支援拖曳上傳或貼上廠商的 510(k) 文件目錄（如 ISO 13485 證書、EMC 測試報告、軟體確效報告等），並整合系統預設指引進行智能比對。
•	強類型結構化輸出 [SRS-F-002-1]： 後端呼叫 geminiService.ts 時，必須強制配置 responseMimeType: "application/json"，並定義嚴格的 responseSchema。AI 返回之 JSON 數據必須嚴格分檢為以下三類項目：
1.	必要項目 (Required)： 經模型比對，該器材類別不可或缺之核心合規文件（如不具備，將導致系統於前端提示「建議立即判定退件」）。
2.	非必要項目 (Not Required)： 自動識別可豁免之文件（如第一等級醫材免附之複雜原物料檢驗單），防止文件冗餘。
3.	選用、建議項目 (Optional)： 非強制但能加速臨床評估或獲得學術優先審查權限之文件。
3.1.3 審查報告生成與雙模式編輯器 [SRS-F-003]
•	功能描述： 系統必須能自動拉取合規審查報告模板，並填充出 3000 至 4000 字的高品質繁體中文與英文 510(k) 實質等同性審查報告。
•	雙模式預覽 [SRS-F-003-1]： 報告生成後，系統必須允許官員一鍵切換於「Markdown 快照預覽（結合主動式 Mermaid 交互圖）」與「原始實體 Markdown 文字編輯器」之間，確保審查官享有 100% 的修改與最終裁決自主權。
3.2 內建六大 WOW AI 功能規格 [SRS-WOW-001 ~ 006]
•	【WOW 1】Mermaid 視覺化審查流程圖自癒引擎： 系統必須在 /src/components/Mermaid.tsx 中置入正則自動修復器：在載入圖表前，自動查找所有方括號、圓括號內的非轉義文字，並以雙引號封裝（例如將 safeChart.replace(/\[([^"\]]+)\]/g, '["$1"]')），將前端 Mermaid 語法崩潰率壓低至 0.2% 以下。
•	【WOW 2】國際法規智能對照： 系統在生成指引時，必須自動調用本地快取的本體庫（Ontology），橫向將本地審查法規條款與美國 FDA Product Code 進行關聯，縱向對應至歐盟醫療器材法規（MDR 2017/745）的分類體系（Class I, IIa, IIb, III）。
•	【WOW 3】模擬退件風險評估： 系統必須比對歷史退件大數據特徵庫，針對特定器械進行關鍵漏洞前警示（如「缺少原廠不含 DEHP 塑化劑宣告書」、「臨床實驗受試者數量不足」）。
•	【WOW 4】法規溯源熱區圖： 系統必須以 GitHub Flavored Markdown (GFM) 表格格式輸出熱區對照表，詳盡勾勒出器械安全性需求與其背後法規來源（如醫療器材管理法條款）的因果關聯。
•	【WOW 5】自動生成補件問答集： 系統必須預測廠商在本次送審中的模糊地帶，自動生成 3 至 5 個極具針對性、用詞客觀但立場堅定的 Q&A 補件問答範例，作為官方公文之藍本。
•	【WOW 6】審查報告語氣與合規性分析： 系統必須針對產出的長篇審查報告進行最後的「元審查 (Meta-Review)」，自動檢測並過濾主觀色彩濃厚之貶低詞彙（如「設計拙劣」），自動校調至中性客觀之法規表述（如「該設計依現有說明書資料尚難佐證具備臨床安全性」）。

4. 三種部署方案評估與深度對比 (Deployment Topology Evaluation)

4.1 方案一：全公有雲部署 (GCP Cloud Run)
•	架構設計： 應用層完全封裝部署於 GCP Cloud Run，AI 運算層直連雲端 Vertex AI 通道。
•	優勢： 具備近乎無限的彈性擴充能力，首字響應極快，免維護本地伺服器硬體；天然具備 2M tokens 的超長文本吞吐能力。
•	劣勢： 廠商送審之未公開軟體專利核心源代碼必須傳輸至雲端端點，存在合規與洩密疑慮。
4.2 方案二：純本地私有化部署 (Local On-Premise Gov-Cloud)
•	架構設計： 系統前端、後端與 AI 推理核心完全常駐部屬在 FDA 內部機房的虛擬化 Hypervisor 與專屬 GPU 伺服器叢集中。物理隔離，與網際網路完全斷絕。
•	優勢： 最高層級之資料隱私保障，送審機密資料 100% 留存本地實體，完全符合最嚴格的政府機關資安工作防護準則。
•	劣勢： 初始硬體採購成本巨大；本地大模型之長文本 Context 處理能力與視覺解析精度，顯著受限於本地 GPU 顯存規模，且相比雲端模型存在約 5% 的推理效能落差。
4.3 方案三：混合雲部署 (Hybrid Model)
•	架構設計： 「資料分級、動態脫敏」。本地伺服器內建置脫敏代理（De-identification Agent），上傳之文檔先於本地執行命名實體識別（NER），抹除一切智慧財產權敏感特徵後，將中性法規條款送上公有雲進行大批量組件生成，再回傳本地進行最終拼裝。
•	優勢： 兼顧雲端模型的超強推理力與本地端隱私安全性。
•	劣勢： 系統整合複雜度極高，需常態維護雲、地兩端模型的狀態同步與資料路由策略。

7. 軟體開發生命週期與建置編譯策略 (SDLC & Build Strategy)
為了保障持續建置、測試及部署在本地全端架構中能夠平穩運行，本系統工程編譯必須遵循以下集成方案：
7.1 靜態類型安全檢測 (Type-Safe Check)
在程式碼進入本地代碼倉庫後，自動化 CI/CD 管線必須立刻觸發 npm run lint 和靜態預編譯檢測。管線必須嚴格掃描全端項目中是否存在任何未命名變數、未處理的異步拋錯（Unhandled Promise Rejection）或多餘的 any 強制轉型宣告，凡未通過類型安全檢測者，管線強制熔斷（Build Failure），拒絕上線。
7.2 esbuild 萬能極速 CommonJS 封裝策略
當後端微服務在本地伺服器中進行產品組建編譯時，package.json 中的 build 腳本必須精確配置為：
Bash
vite build && esbuild server.ts --bundle --platform=node --format=cjs --packages=external --sourcemap --outfile=dist/server.cjs
1.	徹底跨過 Node.js ESM 相對路徑解碼衝突坑： esbuild 引擎會在 50 至 100 毫秒內，將後端服務及其相對引用打包編譯成單一的 dist/server.cjs 檔案。藉由輸出符合 CommonJS 規範的 .cjs 副檔名，Node.js 執行器在冷啟動時無須再理會 package.json 中 "type": "module" 的語法衝突，亦無須理會路徑未加副檔名的解析錯誤。一鍵 node dist/server.cjs 即可秒級啟動，使微服務之冷啟動時間（Cold Start Time）近乎無限逼近 0。
2.	排除外部相依套件膨脹： 藉由配置 --packages=external 參數，強制指示 esbuild 在打包時將 Express 框架、官方 @google/genai SDK 等重量級外部相依套件排除在編譯本體之外，保持 bundle 檔案體積不超過 200KB。外部套件依然由容器底層原生引入，大幅優化部署效率。

8.	問題 8： 在 WOW 9 的動態雷達圖中，如果有多個類似品（Predicates）的臨床試驗基準存在嚴重異質性（例如：一者為大鼠實驗，一者為人體回顧研究），系統應採取何種加權權標（Confidence Weights），才能向審查官發出客觀、具備統計學效力的信心度警告？

9. 總結 (Summary)
本系統需求規格（SRS）暨在地化技術規格說明書，為法規主管機關引進次世代智慧 AI 輔助審查工作站奠定了嚴謹、科學且具備最高合規約束力的技術底座。系統透過高度安全的 TypeScript 全端架構與 esbuild CommonJS 極速打包策略，將本地容器啟動延遲與串流生成阻礙降至最低，並藉由瑞士極簡現代主義介面完美減輕審查官長期查核之視覺疲勞。
在功能層面，系統完美融合了法規更新差分（WOW 7）、多模態包裝標籤合規稽核（WOW 8）與多學科臨床效益-風險鏈式推論（WOW 9），將傳統純文字比對重構為多維度的智慧監管防線。配合極致量化的本地 GPU 顯示卡硬體資產配額與完全物理隔離（Air-Gapped）的傳輸抹除保護機理，本系統提供了一份兼顧運算效能、行政中立與資安最高隱私權益的劃時代智慧型審查工作站落地實施藍圖。
