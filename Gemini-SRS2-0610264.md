智慧型醫療器材（SaMD）510(k) 智慧審查輔助系統——全方位部署架構與契約建置實施計畫書
文件層級： 系統部署架構決策與工程實施計畫書（招標合約專用技術規格附件）
審查機關： 美國食品藥物管理局（US FDA）高階資訊架構監管小組、資訊安全長官（CISO）與採購決策委員會
文件目標： 針對基於 Agentic AI 架構之 510(k) 實質等同性（Substantial Equivalence）智慧審查系統，提供極致詳盡之三種部署方案（全雲端、純本地、混合雲）的拓撲設計、硬體與虛擬化資源清單、網路拓撲、資安防護與災難復原計畫，作為高階官員合約變更與招標驗收之最高法定依據。
1. 導言與全球法規資安背景 (Introduction & Regulatory Context)
智慧醫療器材（SaMD）與包含複雜深度學習模型之智慧輔助診斷系統的送審量逐年翻倍。本智慧審查輔助系統（SmartMed Review System 2.0-060326）之建置，其本質是利用 30 至 35 個獨立運行的 AI 智慧代理（Agentic AI Agents）網絡，執行大批量送審文件目錄強類型分撿、跨國法規本體庫對照、多模態標籤影像（IFU）合規稽核以及鏈式臨床效益-風險權衡（Benefit-Risk Assessment）推論。
由於系統處理之數據涵蓋醫療器械廠商的核心軟體專利、研發代碼、未公開隨機對照試驗（RCT）文獻與病患敏感隱私數據，因此其「部署架構」直接關係到公權力執行的中立性、國家政務資料資安與法律程序溯源（WOW 6 與 WOW 13）。本計畫書針對全雲端（GCP）、純本地私有化（On-Premise）與混合雲（Hybrid）三種拓撲進行極致量化與拆解，供高階長官做出最終戰略裁決。
2. 方案一：公有雲端部署架構計畫 (Google Cloud Platform - GCP)
2.1 架構設計與運作機理
全雲端架構採用無伺服器（Serverless）微服務容器化設計，其核心邏輯是將所有運算能力、大型語言模型（LLM）推理以及儲存完全交由 Google Cloud Platform 託管。前端 React 19 靜態資源經編譯後部署於 Cloud Storage，並透過 Cloud CDN 進行全球邊緣節點加速。
應用服務層（Node.js + Express）封裝為輕量化 Docker 映像檔，運行於 GCP Cloud Run 之上。前端一切與大型語言模型的握手與傳輸流，均由 Express 後端充當安全反向代理（Proxy Service），嚴防 API 金鑰暴露於瀏覽器環境。AI 核心運算層透過安全通道直連 GCP Vertex AI 企業級端點，動態調度旗艦級 Gemini 3.1 Pro（承載長文本臨床文獻與審查報告編寫）與高效能 Gemini 3 Flash（承載清單 JSON 強類型分撿與日誌追補）模型。
2.2 詳細硬體、虛擬化與雲端資源配額清單
為確保系統於高併發（Concurrency）審查情境下維持首字響應延遲（TTFT）小於 350 毫秒之極致體驗，招標合約必須嚴格指定以下配額：
•	運算節點 (GCP Cloud Run - Serverless Container)：
o	虛擬 CPU (vCPU)： 基礎實體配置 2 vCPU，設定自動水平擴展（Auto-scaling）範圍為：最小保留實體數 $1$、最大自動擴展實體數 $50$。
o	系統記憶體 (RAM)： 4 GB DDR5 虛擬分配記憶體。
o	單一實體最大併發限制 (Concurrency)： 80 筆。當單一容器之併發請求數超過 80 時，GCP 底層負載平衡器（Cloud Load Balancing）必須於 1.2 秒內橫向拉起新容器實體。
•	儲存與快取層：
o	Cloud Storage (物件儲存)： 配置 512 GB Standard 儲存空間，用於收納廠商遞交的多模態標籤圖檔、彩色 PDF 說明書及臨時產出之快照。配置「30天自動生命週期抹除政策 (Lifecycle Expiry Policy)」，過期資料物理粉碎。
o	Cloud Memorystore (Redis 快取)： 配置 10 GB 記憶體實體，用以緩存常用法規對照本體庫（MDR/FDA Product Code）及短期 Session 狀態，消除重複調用。
•	網路與安全防護配額 (Networking & Security WAF)：
o	Cloud Armor (邊緣安全防禦)： 內建 OWASP Top 10 防禦策略，配置防 DDoS 惡意流量阻斷、地理位置存取限制。
o	出口頻寬 (Outbound Bandwidth)： 基礎 100 Mbps 至 1 Gbps 彈性頻寬。
•	AI 算力通道 (Vertex AI API Access Rate Limits)：
o	gemini-3.1-pro-preview：配額上限限制為每分鐘請求數（RPM）$1,000$ 次、每分鐘權杖數（TPM）$2,000,000$ 個。
o	gemini-3-flash-preview：配額上限限制為每分鐘請求數（RPM）$5,000$ 次、每分鐘權杖數（TPM）$10,000,000$ 個。
2.3 網路拓撲與安全邊界
全雲端部署採用嚴格的虛擬私有雲（VPC）隔離。Cloud Run 必須配置「輸出 VPC 連線（VPC Connectors）」，使所有後端容器在訪問其他雲端資源時，完全行經內部私有 IP 網段（如 10.128.0.0/20）。禁止任何儲存庫、快取記憶體暴露公共網路端點。所有對外流量統一經由 Cloud NAT 轉發，維持外網 IP 之固定與純淨。
2.4 備援與災難復原方案 (DR & Business Continuity)
採用跨區域雙活（Multi-Region Active-Active）部署。主運算區設於 us-east4（維吉尼亞），副備援區設於 us-central1（愛荷華）。當主區域因雲端機房大規模故障斷線時，全球負載平衡器必須於 500 毫秒內將審查官流量 100% 自動無縫切換至副備援區，維持 99.99% 之系統可用性（SLA）。
3. 方案二：純本地私有化部署架構計畫 (On-Premise Gov-Cloud)
3.1 架構設計與運作機理
純本地私有化部署方案是針對資訊安全防護極度嚴苛、數據 100% 不得離地之隔離內網網路環境（Air-Gapped Gov-Cloud）所設計。系統之基礎軟體技術棧（React 19 + Node.js）完全部屬至 FDA 內部機房的虛擬化 Hypervisor（如 VMware vSphere 8 / Proxmox VE）所拉起的獨立 Ubuntu Server 叢集中。
其最大挑戰在於「AI 運算層」的本地建置。系統拋棄雲端 Google GenAI SDK，改採於本地 GPU 伺服器叢集中載入開源最頂尖之邊緣大模型：文字與推理核心採用 Gemma-2-9B-IT（90億參數微調版）或 Llama-3-8B-Instruct；多模態視覺核對則部屬開源視覺大模型（如 Paligemma-3B 或 Llama-3.2-11B-Vision），並透過 vLLM（高效能推斷加速引擎）封裝為與 OpenAI 相容之內部 REST API 端點，供 Express 後端本地調用。
3.2 詳細硬體、虛擬化與本地資源清單
此方案硬體資產採購成本巨大，為承載大批量審查長文本流（長達 128K 脈絡窗解碼），招標合約之核心硬體規格清單必須包含以下固定資產：
•	高階 AI 推理 GPU 伺服器 (建議採購 2 臺 4U 機架式高密度運算伺服器)：
o	中央處理器 (CPU)： 每臺配置 Dual AMD EPYC 9354（單顆 32 核心、64 執行緒，雙顆共 64 核心、128 執行緒，主頻 3.25GHz）。
o	系統記憶體 (RAM)： 每臺配置 512 GB DDR5 4800MHz ECC REG 企業級記憶體，防範極限高壓運算下之位元翻轉崩潰。
o	圖形加速卡 (GPU Core & VRAM 配置)： 每臺伺服器滿載安裝 4 張 NVIDIA A100 Tensor Core GPU (80GB HBM2e 顯存版本) 或最新 H100 PCIe NVLink 版。單臺伺服器具備高達 320GB 的超高速顯示記憶體，確保能同時將文本大模型與多模態視覺模型常駐（Pinned）於顯示記憶體中。
•	基礎應用伺服器 (承載 Web 前端與 Express 後端微服務)：
o	配置 1 臺 2U 機架式伺服器，Dual Intel Xeon Silver 4410Y（共 24 核心、48 執行緒），128 GB RAM。
•	本地儲存庫與矩陣規格：
o	每臺 GPU 伺服器配置 4 顆 1.92TB Enterprise NVMe SSD 實體硬碟，建立硬體 RAID 10 陣列，維持吞吐速度大於 6000 MB/s。
o	中心儲存：採購 1 臺 24 槽位企業級 NAS 網路儲存設備，內裝 12 顆 16TB 企業級 SATA HDD（配置 RAID 6），提供實質大於 140TB 的持久化數據儲存空間，用以儲存歷史案件。
•	網路交換設備：
o	採購 1 臺 24 埠 10 Gbps (萬兆) 光纖乙太網路交換機，以及 1 臺 Mellanox 100 Gbps InfiniBand 交換機（用於 GPU 伺服器間之 NVLink 跨機高速通訊，消除模型多卡分散式推理時的頻寬瓶頸）。
3.3 網路拓撲與安全邊界
純本地部署採取物理隔離（Air-Gapped）。系統所處網路區域定義為「FDA 核心安全控制區（Zone 1）」。系統伺服器嚴禁連接公共網際網路（Internet），任何外部修補檔、法規庫更新，必須由高階系統管理員經由專屬防毒堡壘機（Jumpbox）透過雙重認證（2FA）手動審查後，以專用加密隨身碟（U盤）導入。內部網絡實施精細的 VLAN 劃分：Web 服務區（VLAN 10）、 Express 後端代理區（VLAN 20）與 GPU 運算叢集區（VLAN 30），各區域間架設硬體防火牆（如 Fortinet），強制執行嚴格的埠號（Port）白名單控制。
3.4 備援與災難復原方案 (DR & Business Continuity)
兩臺 AI 伺服器配置為本地高可用容錯叢集（High-Availability Cluster）。前端請求透過硬體 F5 負載平衡器進行輪詢（Round-Robin）分流。當伺服器 A 發生硬體故障（如電源毀損、硬碟陣列離線）時，負載平衡器立即剔除該節點，伺服器 B 承接全量審查請求，本地系統在 3 秒內自動恢復。數據備份實施「本地-異地」3-2-1 備份策略，每夜間自動將稽核日誌與加密快照同步至 FDA 備份機房。
4. 方案三：混合雲部署架構計畫 (Hybrid Topology)
4.1 架構設計與運作機理
混合雲部署方案是在「極致推理效能（雲端 Gemini 旗艦模型之 2M 超長 Context 與多模態外觀排版識別精度）」與「極限隱私資安（醫療器械廠商之核心智慧財產權與專利代碼不離地）」之間，採取高度精緻、可動態調度、最具工程可行性的拓撲設計。
系統採取「資料分級、動態脫敏、運算分流」的核心機理。Web 用戶端與 Express 後端微服務部署於 FDA 本地機房中。
當審查官員上傳 510(k) 申報文件時，本地 Express 後端首先觸發「本地安全去識別化與脫敏代理（De-identification Agent）」。此代理利用正則表達式、具備命名實體識別（NER）的輕量化本地 AI 模型，自動過濾並剔除所有涉及廠商專利敏感參數、代碼底層邏輯與敏感個人資料（PII）。
處理後的「非涉密、已脫敏」文本、全球通用法規對照、公眾標準文本，透過安全、高強度加密的 VPN 專線，調度雲端 GCP Vertex AI 之 Gemini 3.1 Pro 進行全自動化超長文本摘要與 4000 字審查報告草稿生成。而未脫敏之原始技術文檔、軟體架構源代碼、影像特徵圖，則留存於本地內網，由輕量化本地模型執行敏感詞抽樣與基礎結構查核。最後，雲端回傳之中性報告草稿與本地稽核結果於本地前端 React 進行最終組裝與 Mermaid 可視化圖表自癒渲染。
4.2 詳細硬體、虛擬化與混合雲資源清單
混合雲方案大幅精簡了本地 GPU 的採購規模，將百萬級硬體支出轉向雲端服務費用：
•	本地端硬體資源配置 (1 臺 2U 機架式輕量 AI 伺服器)：
o	中央處理器 (CPU)： Single AMD EPYC 9124（16 核心、32 執行緒）。
o	系統記憶體 (RAM)： 128 GB DDR5 ECC 記憶體。
o	圖形加速卡 (GPU 配置)： 僅需配置 2 張 NVIDIA RTX 4090 (24GB VRAM) 或 2 張 NVIDIA A40 (48GB VRAM)。此顯存規模足以順暢載入、常駐 7B-13B 等級之脫敏與 PII 過濾模型，免去購置百萬級 H100 叢集之預算。
o	本地儲存： 2TB NVMe SSD，加 1 臺 8 盤位 NAS 作為案件備份。
•	雲端 GCP 資源預留配額：
o	同方案一之 Cloud Run 輕量配置（設定最大擴展實體數為 15 即可，僅用於橋接雲端 API，非託管核心 Web 站點）。
o	預留 Vertex AI API 流量，配額調降為全雲端之 50%，大幅節省預算。
4.3 網路拓撲與安全邊界
混合雲架構之網路設計最為精緻。本地 FDA 機房與 GCP VPC 之間，必須建置一條常態維持之「雲端專線連結（Cloud Interconnect）」或架設高強度硬體 IPSec VPN 隧道。網路傳輸強制實施最高規格之端到端加密（TLS 1.3 且限定密碼學套件如 TLS_AES_256_GCM_SHA384）。GCP 端設置內部負載平衡器（Internal Load Balancer），只允許行經 VPN 專線進入的 FDA 內部固定 IP 請求存取 Vertex AI 模型端點，外網公共網際網路完全無法對此內部端點進行任何偵測或掃描。
4.4 備援與災難復原方案 (DR & Business Continuity)
混合雲架構具備極佳的降級容錯特性（Graceful Degradation）。當雲端 VPN 專線意外遭物理挖斷或 GCP 雲端服務發生大規模中斷時，本地 Express 後端代理會自動偵測到連線超時（Timeout > 5 秒）。系統會立即觸發「邊緣退守控制機制（Edge Fallback Mechanism）」：於 UI 終端機向審查官拋出黃色預警日誌，並將審查推理引擎一鍵切換至本地輕量化模型運行。雖然此時模型的報告長度會從 4000 字縮減至 1500 字，且長文本 Context Window 縮減，但仍能確保系統「業務不中斷、全天候可點擊」。

6.1 功能性需求項目 (Functional Requirements)
•	[SRS-F-001] 智慧指引生成與串流核心： 系統後端必須使用最新 @google/genai 官方 SDK 架構，嚴禁使用已廢棄之舊版庫。大文本指引生成時，強制採取前端 for await...of 異步迭代器。
o	驗收指標： 首字渲染延遲（TTFT）在雲端模式下必須 $\le 350$ 毫秒，且前端遙測終端機必須能 100% 攔截 AbortSignal，點擊中斷時立刻在 100 毫秒內終止後端 TCP 連線以停止 Token 耗用。
•	[SRS-F-002] 強類型結構化送審清單分撿： 模型在進行文件目錄比對時，Express 後端發送之提示詞參數必須內嵌 responseMimeType: "application/json"，且必須配置嚴格的強類型 responseSchema（詳見招標附件代碼段）。AI 返回之數據必須精確落入 required、not_required 及 optional 三大陣列中。

7. 軟體開發生命週期、測試與部署工程實施策略 (SDLC & CI/CD Lifecycle)
為保障持續建置、測試及部署在 AI 微服務架構中能夠平穩運行，招標合約指定實施以下極速封裝與整合策略：
7.1 靜態類型安全檢測管線 (Type-Safe Pipeline)
在工程代碼提交（Git Push）至原始碼倉庫後，自動化 CI/CD 管線必須立刻觸發 npm run lint 和靜態預編譯檢測。管線必須嚴格掃描全端項目中是否存在任何未命名變數、未處理的異步拋錯（Unhandled Promise Rejection）或多餘的 any 強制轉型宣告，凡未通過類型安全檢測者，管線強制熔斷（Build Failure），拒絕上線。
7.2 esbuild 萬能極速 CommonJS 封裝策略
當後端服務在 Full-Stack 容器或本地伺服器中進行產品組建編譯時，package.json 中的 build 腳本必須精確配置為：
JSON
"scripts": {
  "build:backend": "esbuild server.ts --bundle --platform=node --format=cjs --packages=external --sourcemap --outfile=dist/server.cjs"
}
1.	徹底跨過 Node.js ESM 相對路徑解碼衝突坑： esbuild 引擎會在 50 至 100 毫秒內，將後端服務及其錯綜複雜的相對引用（Relative Imports）打包編譯成單一的 dist/server.cjs 檔案。藉由輸出符合 CommonJS 規範的 .cjs 副檔名，Node.js 執行器在冷啟動時無須再理會 package.json 中 "type": "module" 的語法衝突，亦無須理會第三方舊模組路徑未加 .ts/.js 的解析錯誤。一鍵 node dist/server.cjs 即可秒級啟動，使雲端微服務之冷啟動時間（Cold Start Time）近乎無限逼近 0。
2.	排除外部相依套件膨脹 (Bundle Swelling)： 藉由配置 --packages=external 參數，強制指示 esbuild 在打包時將 Express 框架、官方 @google/genai SDK 等重量級外部相依套件排除在編譯本體之外，保持 bundle 檔案體積不超過 200KB。外部套件依然由容器底層原生引入，大幅優化容器映像檔之分層快取（Layer Caching）速度與部署效率。
