智慧型醫療器材（SaMD）510(k) 智慧審查輔助系統——系統需求規格書與招標契約決策報告
文件層級： 招標合約暨核心系統規格書（供高階決策官員審閱）
審查機關： 美國食品藥物管理局（US FDA）高階資訊暨法規決策小組
文件目標： 確立智慧型代理（Agentic AI）510(k) 審查系統之系統需求規格（SRS）與部署資源配置，作為未來政府合約簽署之最高技術範本。
1. 引言與核心願景 (Introduction & Core Vision)
1.1 系統核心願景
隨著醫療器材軟體（SaMD）以及內嵌 AI 演算法之跨領域複合醫材申請案件量呈指數級增長，傳統實質等同性（Substantial Equivalence）審查面臨兩大核心瓶頸：送審技術文件結構極度繁瑣，以及法規條文與廠商自主研發數據之間存在巨大的「語意鴻溝」。
本報告所確立之次世代智慧醫材審查工作站（SmartMed Review System 2.0-060326），旨在為 FDA 高階審查官員（Reviewing Officers）提供一套「Agentic AI 增強型智權審查輔助工作站」。本系統利用多個自主運行的 AI 代理（Agents）網絡，將枯燥的法規條文與數百頁的申報書自動轉化為可視化的審查地圖、自動化文件分類矩陣與結構化審查報告，旨在將每案平均查核時間縮短 40% 以上，同時維持行政機關科學客觀之公權力防線。
1.2 高階資訊架構與瑞士極簡美學
為確保高階審查官員在長時間、高強度的閱讀環境中維持最高專注度，系統必須嚴格遵循瑞士極簡現代主義（Swiss Modernism）視覺語彙：
•	高對比度底色配置： 系統應預設提供極簡明亮主題（Soft Off-Whites 與 Deep Charcoal Grays）與深邃暗色模式（Zinc 50 與 Zinc 950），確保在任何環境光線下眼睛疲勞度降至最低。
•	比例化負空間（Negative Space）： 頁面佈局完全捨棄繁雜的多級導航與干擾視線的進度條，主視窗全面配置具備呼吸感的網格寬度限制（max-w-6xl mx-auto），使高階官員的注意力集中於核心審查與比對區塊。
•	字型美學與資訊對齊： 標題採用高科技幾何質感的 Inter 搭配 Outfit 無襯線英文字型；涉及法規條文、程式碼區塊與系統遙測日誌（Telemetry Logs）之區域，全面強制配置 JetBrains Mono 等寬字型，以實現嚴格的資訊視覺對齊。
2. 核心技術棧規格 (Technical Stack Specifications)
本系統基於高可靠、低延遲架構構建，完全採用 TypeScript 語言實現端到端的類型安全（Type-Safety）。其三層架構定義如下：
1.	瀏覽器用戶端 (前端 React 19)：
o	利用並行渲染（Concurrent Rendering）特性，在大規模 Markdown 與圖表生成時維持 UI 響應度。
o	建置工具採用 Vite 6，配合原生 ES 模組（ESM）解析提供毫秒級組建反饋。
o	樣式採用 Tailwind CSS 4 引擎，全面優化並縮減靜態樣式表體積。
o	整合 Mermaid 版本 11 進行動態法規路徑渲染，並導入 react-markdown 與 remark-gfm 渲染高度結構化的表格與文本。
2.	應用服務層 (後端 Node.js + Express)：
o	用於代理、守護和橋接高階 Google GenAI API，防止 API 金鑰暴露於前端環境，並統一記錄稽核日誌。
o	執行環境採用聯邦隔離之容器（Cloud Run / Docker Container），與 Nginx 反向代理整合，統一進行路由與證書安全防護。
3.	雲端/本地 AI 運算層 (Google GenAI API / 邊緣模型)：
o	核心驅動採用最新 @google/genai TypeScript SDK，支援 Gemini 3.1 Pro / Flash 等具備強類型結構化輸出（Structured JSON Output）能力之模型。
3. 系統需求規格書項目 (Detailed System Requirements Specification - SRS)
為確保採購合約具備高度的法律與技術約束力，以下將系統功能與非功能需求以標準 SRS 規格進行拆解與量化定義。
3.1 功能需求規格 (Functional Requirements)
3.1.1 智慧指引生成模組 [SRS-F-001]
•	功能描述： 系統必須允許審查官員上傳 .txt 或 .md 格式的異質法規原始草案，或者由審查官手動貼上法規核心文本，系統將自動重構為標準化審查指引。
•	串流響應效能 [SRS-F-001-1]： 系統全面整合 generateTextStream 推理架構。透過 for await...of 非同步迭代器，即時將模型輸出的權杖（Tokens）以字元流形式渲染於瀏覽器介面上，首字響應時間（TTFT）必須小於 350 毫秒。
•	畫家擬真風格設定（Painter Personas）[SRS-F-001-2]： 系統必須內建多元思考維度切換器。透過將 System Instruction 微調為「達文西（精準解剖與力學拆解）」、「梵谷（醫學人文關懷與患者舒適度）」、「林布蘭（Revolutionary 功效與失效風險的強烈明暗對比）」、「畢卡索（多維解構法規要求）」等世界大師的藝術哲學，將枯燥條文重構為具備穿透力的論述，引導官員從不同維度審視器材設計。
•	非同步中斷機制 [SRS-F-001-3]： 前端必須配置即時遙測終端機。當審查官按下「立即中斷 (Abort)」功能時，系統必須透過對前端傳遞的 AbortSignal 進行監聽，立刻向 Gemini 伺服器發送 TCP 終止框架，全面中斷 Token 消耗。
3.1.2 送審清單智慧分類與強類型分撿模組 [SRS-F-002]
•	功能描述： 系統必須提供雙向大塊文本對照視窗。左側支援拖曳上傳或貼上廠商的 510(k) 文件目錄（如 ISO 13485 證書、EMC 測試報告、軟體確效報告等），並整合系統預設指引進行智能比對。
•	強類型結構化輸出 [SRS-F-002-1]： 後端呼叫 geminiService.ts 時，必須強制配置 responseMimeType: "application/json"，並定義嚴格的 responseSchema。AI 返回之 JSON 數據必須嚴格分檢為以下三類項目：
1.	必要項目 (Required)： 經模型比對，該器材類別不可或缺之核心合規文件（如不具備，將導致系統於前端提示「建議立即判定退件」）。
2.	非必要項目 (Not Required)： 自動識別可豁免之文件（如第一等級醫材免附之複雜原物料檢驗單），防止文件冗餘。
3.	選用、建議項目 (Optional)： 非強制但能加速臨床評估或獲得學術優先審查權限之文件。
3.1.3 審查報告生成與雙模式編輯器 [SRS-F-003]
•	功能描述： 系統必須能自動拉取合規審查報告模板，並填充出 3000 至 4000 字的高品質繁體中文與英文 510(k) 實質等同性審查報告。
•	雙模式預覽 [SRS-F-003-1]： 報告生成後，系統必須允許官員一鍵切換於「Markdown 快照預覽（結合主動式 Mermaid 交互圖）」與「原始實體 Markdown 文字編輯器」之間，確保審查官享有 100% 的修改與最終裁決自主權。
3.2 內建六大 WOW AI 功能規格 [SRS-WOW-001 ~ 006]
•	【WOW 1】Mermaid 視覺化審查流程圖自癒引擎： 系統必須在 /src/components/Mermaid.tsx 中置入正則自動修復器：在載入圖表前，自動查找所有方括號、圓括號內的非轉義文字，並以雙引號封裝（例如將 safeChart.replace(/\[([^"\]]+)\]/g, '["$1"]')），將前端 Mermaid 語法崩潰率壓低至 0.2% 以下。
•	【WOW 2】國際法規智能對照： 系統在生成指引時，必須自動調用雲端本體庫（Ontology），橫向將本地審查法規條款與美國 FDA Product Code 進行關聯，縱向對應至歐盟醫療器材法規（MDR 2017/745）的分類體系（Class I, IIa, IIb, III）。
•	【WOW 3】模擬退件風險評估： 系統必須比對歷史退件大數據特徵庫，針對特定器械進行關鍵漏洞前置預警（如「缺少原廠不含 DEHP 塑化劑宣告書」、「臨床實驗受試者數量不足」）。
•	【WOW 4】法規溯源熱區圖： 系統必須以 GitHub Flavored Markdown (GFM) 表格格式輸出熱區對照表，詳盡勾勒出器械安全性需求與其背後法規來源（如醫療器材管理法條款）的因果關聯。
•	【WOW 5】自動生成補件問答集： 系統必須預測廠商在本次送審中的模糊地帶，自動生成 3 至 5 個極具針對性、用詞客觀但立場堅定的 Q&A 補件問答範例，作為官方公文之藍本。
•	【WOW 6】審查報告語氣與合規性分析： 系統必須針對產出的長篇審查報告進行最後的「元審查 (Meta-Review)」，自動檢測並過濾主觀色彩濃厚之貶低詞彙（如「設計拙劣」），自動校調至中性客觀之法規表述（如「該設計依現有說明書資料尚難佐證具備臨床安全性」）。
3.3 前瞻性多模態與臨床推論功能規格 [SRS-WOW-007 ~ 009]
3.3.1 【WOW 7】智慧法規更新即時差分與預警引擎
•	功能描述： 當新版國際法規標準上傳時，系統後端必須啟動 Vector-Diff 引擎。利用 Gemini 嵌入模型（Embedding API）將文本切片並映射至高維向量空間，透過餘弦相似度捕捉語意變更。
•	前端呈現： 必須以紅（刪除語段）、綠（新增語段）以及黃（概念性增補）呈現差異，並主動提示新法條對既有審查指引的衝擊等级（Impact Level: High/Medium/Low）。
3.3.2 【WOW 8】多模態醫材標籤與說明書外觀視覺合規稽核
•	功能描述： 系統必須支援審查官上傳醫材外觀包裝、彩色標籤 PDF 或使用說明書（IFU）圖檔。
•	視覺物件偵測與 OCR 解析： 利用 Gemini Multimodal 多模態技術，一方面提取圖像中微小的免責聲明、批號等文字，另一方面識別生物危害圖章、滅菌狀態圖示、CE/TFDA 規範標章。
•	排版合規性查核： 系統必須自動判斷「警示文字之色彩對比度是否滿足 WCAG 與 FDA 標準（字體是否小於 6pt）」及「是否遺漏法定製造商地址或代理商登錄字號」，並於前端交互式 Canvas 上以黃色/紅色數位方框標註「缺失或疑慮區域」。
3.3.3 【WOW 9】智慧型動態多學科臨床效益-風險權衡推論引擎
•	功能描述： 當審查官上傳廠商檢附的臨床文獻或 RCT 試驗報告摘要時，AI 必須深入提取所有的功效指標（主要臨床終點對照組與實驗組之差異、Sensitivity、Specificity、AUC）與不良事件發生率（併發症機率、系統故障機率）。
•	鏈式提示詞推理 (Chain-of-Thought)： 利用模型自我調校的推理鏈，推演「若該醫材具有較高的不良事件發生率，其帶來的臨床效益是否顯著大於現有療法？是否有合理的限制使用對象或加註警語之配套方案？」，並藉由 D3.js / Recharts 自動生成多維雷達平衡對焦圖與中立的臨床風險平衡結論建議。
3.4 非功能需求規格 (Non-Functional Requirements)
3.4.1 安全性與法規隱私保障 [SRS-SEC-001]
•	金鑰防護： 核心 API 金鑰必須完全隔離於 Cloud Run 或私有化伺服器之環境變數（Environment Variables）中，前端一切調用必行 Express 端點代理轉發，嚴禁於客戶端暴露。
•	原始專利零留存政策 (Zero-Retain)： 系統與 Google Enterprise APIs 對接時，必須預設啟用免於二次訓練協定。後端發送之內容加載預設不開啟歷史紀錄，且不進行本地資料庫暫存，嚴防醫療器材廠商的核心軟體專利與研發機密外洩。
•	傳輸加密： 反向代理伺服器（Nginx）上必須強制實施嚴格的 TLS 1.3 協定握手，拒絕一切不安全的 HTTP 請求，徹底防範中間人（MITM）嗅探。
•	本地暫存抹除： 前端僅保留 Session 層級狀態，當頁面點擊重置或刷新時，本地狀態即刻抹除，防止多名審查官於共享工作站上瀏覽到前一案之送審機密。
3.4.2 系統效能與可用性 [SRS-PER-001]
•	併發處理能力： 系統架構必須在應對大批量審查官員高併發使用時，透過優化之快取與路由策略，防止出現瞬時 429 API Rate Limit 超限。
•	靜態 bundle 體積限制： 前端打包編譯時，必須剝離多餘的遺留組件體積，將生產環境的靜態樣式表與核心 bundle 控制在極低體積（CSS 限制於 45KB 以內），以實現毫秒級的冷啟動。
4. 三種部署方案評估與詳細資源需求 (Deployment Options & Resource Requirements)
針對 FDA 資訊安全防護等級與行政審查網路環境的不同（政務外網 vs. 隔離內網），合約招標必須就以下三種部署方案（GCP 雲端、純本地 On-Premise、混合雲 Hybrid）進行嚴格的技術、預算與硬體資源配置對焦。
4.1 方案一：全雲端部署 (Google Cloud Platform)
4.1.1 架構與運作機理
應用服務層完全封裝為 Docker 映像檔，部署於 GCP Cloud Run（無伺服器容器服務），前端靜態資源託管於 Cloud Storage 並透過 Cloud CDN 加速。安全防護端由 Cloud Armor 提供 WAF 與抗 DDoS 防禦。AI 運算層則直接調用 Google Vertex AI 企業級 API 通道。
4.1.2 詳細硬體/虛擬化資源需求
•	運算節點 (Cloud Run 配置)：
o	CPU： 每個實體配置 2 vCPU（可自動水平擴展至最大 50 個實體）。
o	記憶體 (RAM)： 每個實體配置 4 GB RAM。
o	最大併發數限制： 每個實體設定最大併發請求數（Concurrency）為 80。
•	網路頻寬： 基礎配置 100 Mbps 出口頻寬（具備自動擴展彈性）。
•	後端儲存： Cloud Storage 500 GB（用於高階多模態上傳圖檔與暫存報告之臨時快照）。
•	AI 算力支援： 雲端代管之無限制 Gemini 3.1 Pro/Flash API 存取權限。
4.1.3 優缺點深度剖析
•	優勢： 具備近乎無限的彈性擴充能力，高併發審查時模型完全不卡頓；模型版本升級由雲端代管，零維護成本；完整支援超長上下文（Up to 2M tokens），可一次性吞吐數百頁之臨床試驗報告。
•	劣勢： 廠商送審之核心專利或未公開臨床數據必須傳輸至雲端端點，若未配置企業級隱私協定，存在合規疑慮。
4.2 方案二：純本地私有化部署 (Local On-Premise)
4.2.1 架構與運作機理
系統完全部署於 FDA 內部高度隔離之政務機箱或 Gov-Cloud 隔離內網中。硬體層需架設專屬 GPU 伺服器叢集，後端服務與 AI 運算層完全運行於本地內網沙箱。AI 核心需採用開源之邊緣端大模型（如 Gemma-2-9B-IT 或 Llama-3-8B-Instruct），並進行本機封裝。
4.2.2 詳細硬體/虛擬化資源需求
•	硬體伺服器 (建議配置 2 臺 4U 機架式 AI 伺服器)：
o	中央處理器 (CPU)： 每臺配置 Dual AMD EPYC 9354 (共 64 核心，128 執行緒)。
o	系統記憶體 (RAM)： 512 GB DDR5 ECC RAM。
o	圖形處理器 (GPU 算力核心)： 每臺伺服器配置 4 張 NVIDIA A100 (80GB VRAM) 或 H100 NVLink 顯示卡，用以承載本地大模型與多模態 OCR 之並行推理。
•	本地儲存： 4TB Enterprise NVMe SSD (RAID 10配置)，用以儲存本地模型權重（Weights）與全量本地審查稽核日誌。
•	內網網路： 10 Gbps 萬兆光纖乙太網路交換機。
4.2.3 優缺點深度剖析
•	優勢： 最高層級之資料隱私保障，送審機密資料 100% 物理隔離不離地，完全符合最嚴格的監管法規防護準則。
•	劣勢： 硬體採購與電力能耗成本極高；邊緣端本地大模型之長文本處理能力（通常受限於 32K - 128K Tokens）與多模態視覺解析精度顯著低於雲端旗艦模型，且存在約 5% 的分類效能落差。
4.3 方案三：混合雲部署 (Hybrid Topology)
4.3.1 架構與運作機理
採取「資料分級、動態脫敏」策略。系統後端應用服務層與基礎文件去識別化（De-identification）模組部署於本地輕量級伺服器上。非涉密/已去識別化任務（如全球通用法規對照、公眾標準差分、標籤圖像 OCR）透過安全端點轉發給 GCP Vertex AI 雲端旗艦模型處理；而核心涉密專利數據、原始代碼與特定病患隱私臨床數據則留存於本地，由輕量化本地模型執行敏感詞抽樣與初步稽核。
4.3.2 詳細硬體/虛擬化資源需求
•	本地邊緣端資源配置 (1 臺 2U 機架式伺服器)：
o	CPU： Single AMD EPYC 9124 (16 核心)。
o	記憶體 (RAM)： 128 GB RAM。
o	GPU 配置： 2 張 NVIDIA RTX 4090 (24GB VRAM) 或 A40，僅用於本地敏感數據過濾與輕量級文字比對。
o	本地儲存： 2TB SSD。
•	雲端對接資源配置： 同方案一之 Cloud Run 輕量配置（最小實體數設為 1，做為動態流量橋接）。
•	網路安全組件： 需部署專屬之硬體防火牆，常態維持具備高強度加固（TLS 1.3）的專線或安全 VPN 隧道。
4.3.3 優缺點深度剖析
•	優勢： 在「運算效能（利用雲端旗艦模型之超強推理力與大 Context）」與「資料安全（核心專利不離地）」之間取得最佳平衡，極具邊際效益。
•	劣勢： 系統整合複雜度極高，需動態維護本地與雲端兩套模型的 System Prompt、資料路由策略與狀態同步。

本系統需求規格（SRS）暨部署決策報告，為 FDA 未來五年內引進 Agentic AI 輔助 510(k) 審查系統確立了清晰、嚴謹且可量化的技術與招標基準。系統透過高度安全的 TypeScript 全端架構與優化之 esbuild 編譯策略，將容器冷啟動與串流延遲降至極致，並藉由瑞士極簡現代主義介面大幅減輕高階官員的認知負荷。
在核心功能上，系統完美融合了法規即時差分（WOW 7）、多模態包裝外觀稽核（WOW 8）與多學科臨床效益-風險鏈式推論（WOW 9），全面拉高技術招標之防線。高階決策長官可根據機關預算、IT 維護量能以及對資料隱私之極限要求，在「彈性無限的全雲端方案」、「100% 物理隔離的純本地方案」或「兼顧效能與去識別安全的混合雲拓撲」之間做出最符合國家利益之最優戰略抉擇。
